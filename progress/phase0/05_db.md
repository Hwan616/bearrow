## 주요 결정

- **커스텀 마이그레이션 러너 선택**: drizzle-kit의 번들 마이그레이션 방식 대신, `MIGRATIONS` 배열을 코드에서 직접 관리하는 단순 방식 채택. 이유: 초기 단계에서 설정 복잡도 최소화, 마이그레이션 이력이 DB에 직접 기록되어 디버깅 용이.
- **`commonColumns` 헬퍼 분리**: CONVENTIONS의 "모든 레코드는 id·updated_at 필수" 규칙을 강제하기 위해 공통 컬럼을 헬퍼로 추출. Phase 1 테이블 정의 시 `{ ...commonColumns, ... }` 형태로 확장.
- **테스트에서 네이티브 모듈 미사용**: `expo-sqlite` 직접 호출 대신 `SQLiteDatabase` 인터페이스를 모킹하여 순수 JS 테스트 유지.

## 남은 과제 / 주의사항

- **앱 시작 시 `runMigrations` 호출 필요**: Phase 1 첫 테이블 추가 시 `App.tsx`나 앱 초기화 훅에서 `runMigrations(sqliteDb)` 호출 코드 추가.
- **Phase 1.1에서 할 일**: `MIGRATIONS` 배열에 `events` 테이블 생성 SQL 추가, `schema.ts`에 events 테이블 export.
- `drizzle-orm`의 `^` 버전 범위 — 메이저 업그레이드 시 expo-sqlite 드라이버 호환성 확인 필요.
